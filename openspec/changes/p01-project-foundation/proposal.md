# Proposal: p01-project-foundation

## Intent

DAF Local cannot be built or tested without a project skeleton. Before any agent, crew, or token logic is implemented, the Python project must exist with its dependency declarations, CLI entry point, and directory structure in place.

This change creates the foundational scaffolding that every subsequent proposal `(P02–P19)` plugs into. Without it, there is no `daf` command to invoke, no `src/daf/` package to extend, and no `pyproject.toml` to install from.

## Scope

### In scope

- Python project with `pyproject.toml` declaring core runtime dependencies: CrewAI, Anthropic SDK, and a CLI framework (Click or Typer)
- `daf` CLI entry point registered as a console script via `pyproject.toml`
- `daf init` subcommand stub with routing skeleton (no interview logic — that is P02)
- `daf init --profile <path>` flag for non-interactive mode passthrough (§13.5)
- `daf init --resume <path>` flag for resuming interrupted runs (§13.1)
- Project layout: `src/daf/` package with `__init__.py`, `cli.py`, and empty `crews/`, `agents/`, `tools/` subdirectories
- Developer `README.md` with environment setup, installation steps, and first-run instructions
- Optional: `.python-version`, `.env.example`, and basic `Makefile` targets (`install`, `dev`, `lint`)

### Out of scope

- Interview logic and 11-step question flow (P02)
- Any agent or crew implementation (P03–P17)
- CrewAI Crew/Agent/Task definitions
- Token generation, spec generation, or pipeline configuration
- Testing framework configuration beyond what ships with the project skeleton (e.g., `vitest.config.ts` is generated by P08, not here)

## Affected Crews & Agents

No crews or agents are implemented in this change. The skeleton is the prerequisites layer that DS Bootstrap Crew agents will be added to in P02–P09.

## Approach

1. **Initialize the Python project** using `pyproject.toml` with `[project]` metadata, `[project.scripts]` pointing `daf = "daf.cli:app"`, and a `[project.optional-dependencies]` dev group for linting/type-checking.
2. **Choose a CLI framework** — Typer is preferred for its type-annotation-driven argument parsing and automatic `--help` generation; Click is acceptable if Typer introduces incompatibilities.
3. **Scaffold `src/daf/`** with `__init__.py` (version constant), `cli.py` (Typer app + `init` command stub), and empty subdirectories: `crews/`, `agents/`, `tools/`.
4. **Register flags on `daf init`**: `--profile` (optional path, str) and `--resume` (optional path, str); each prints a "not yet implemented" notice and exits cleanly at this stage.
5. **Write `README.md`** covering: prerequisites (Python ≥3.11, Node, API key), `pip install -e .` dev install, first `daf init` run, and a pointer to environment variable configuration (`.env.example`).
6. **Validate the skeleton** by running `daf --help` and `daf init --help` to confirm the CLI registers and prints usage without errors.

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Typer version conflict with CrewAI's own Click/Typer pins | Medium | Pin dependencies explicitly; run `pip install -e .` in CI to catch conflicts early |
| `src/` layout vs flat layout tooling friction | Low | Use `src/` layout consistently — it's the recommended Python packaging pattern and avoids import shadowing issues |
| Future agents require a different package structure | Low | Empty `crews/`, `agents/`, `tools/` directories are placeholders — structure is intentionally shallow and can be deepened in later proposals without breaking this foundation |
