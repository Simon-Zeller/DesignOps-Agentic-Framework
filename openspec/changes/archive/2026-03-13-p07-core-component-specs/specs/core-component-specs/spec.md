# Specification

## Purpose

This spec governs the `CoreComponentSpecGenerator` tool and the Agent 4 (Core Component Agent) module. It defines the behavioral requirements for generating canonical `*.spec.yaml` files for all non-primitive components across the three scope tiers (Starter, Standard, Comprehensive) as produced by the DS Bootstrap Crew (Agent 4, Task T4, §4.1).

---

## Requirements

### Requirement: Scope-tier component coverage

The `CoreComponentSpecGenerator` tool SHALL generate canonical spec YAMLs for the exact component set defined by the requested scope tier (§4.1). The DS Bootstrap Crew's Agent 4 (Core Component Agent) drives this tool as part of Task T4.

- `starter` → 10 components: Button, Input, Checkbox, Radio, Select, Card, Badge, Avatar, Alert, Modal
- `standard` → Starter 10 + 9 more: Table, Tabs, Accordion, Tooltip, Toast, Dropdown, Pagination, Breadcrumb, Navigation
- `comprehensive` → Standard 19 + 7 more: DatePicker, DataGrid, TreeView, Drawer, Stepper, FileUpload, RichText

#### Acceptance Criteria

- [ ] `generate_component_specs(scope="starter", output_dir=...)` writes exactly 10 YAML files
- [ ] `generate_component_specs(scope="standard", output_dir=...)` writes exactly 19 YAML files
- [ ] `generate_component_specs(scope="comprehensive", output_dir=...)` writes exactly 26 YAML files
- [ ] The function returns a `dict[str, str]` mapping PascalCase component names to absolute file paths
- [ ] Calling the function twice with the same arguments is idempotent — files are overwritten without error
- [ ] The `specs/` subdirectory is created automatically if it does not exist
- [ ] Passing an invalid scope value raises `ValueError` with a descriptive message

#### Scenario: Starter scope generates 10 files

- GIVEN an empty temporary directory `output_dir`
- WHEN `generate_component_specs(scope="starter", output_dir=str(output_dir))` is called
- THEN exactly 10 YAML files exist under `output_dir/specs/`
- AND the returned dict has exactly 10 keys matching Starter component names
- AND no Standard-only or Comprehensive-only spec files exist

#### Scenario: Standard scope accumulates Starter components

- GIVEN an empty temporary directory `output_dir`
- WHEN `generate_component_specs(scope="standard", output_dir=str(output_dir))` is called
- THEN 19 YAML files exist under `output_dir/specs/`
- AND `Button.spec.yaml` is present (inherited from Starter tier)
- AND `Table.spec.yaml` is present (Standard-delta component)

#### Scenario: Invalid scope raises ValueError

- GIVEN an invalid scope string `"enterprise"`
- WHEN `generate_component_specs(scope="enterprise", output_dir=...)` is called
- THEN a `ValueError` is raised
- AND the error message mentions the valid values: `starter`, `standard`, `comprehensive`

### Requirement: Required YAML fields per component spec

Every component spec YAML generated by the DS Bootstrap Crew (Agent 4, Task T4) SHALL contain the following top-level fields: `component`, `description`, `props`, `variants`, `states`, `tokenBindings`, `compositionRules`, `a11yRequirements`. These fields are consumed by the Design-to-Code Crew (Phase 3, Agents 12–16) and the Component Factory Crew (Agents 17–20) as their canonical input contract (§3.6).

#### Acceptance Criteria

- [ ] Each YAML contains `component` (string, PascalCase)
- [ ] Each YAML contains `description` (non-empty string)
- [ ] Each YAML contains `props` (object; each key has at minimum `type`, `required`, and `default` sub-fields)
- [ ] Each YAML contains `variants` (non-empty array of strings)
- [ ] Each YAML contains `states` (object; at minimum `default` key; interactive components have `hover`, `focus`, `disabled`)
- [ ] Each YAML contains `tokenBindings` (non-empty array for all non-layout components)
- [ ] Each YAML contains `compositionRules` (object; with `composesFrom`, `allowedChildren`, `forbiddenNesting`, `slots`)
- [ ] Each YAML contains `a11yRequirements` (object; with `role`, `focusable`, `keyboardInteractions`, `ariaAttributes`)

#### Scenario: Button spec structure

- GIVEN `generate_component_specs(scope="starter", ...)` has been called
- WHEN `Button.spec.yaml` is loaded with `yaml.safe_load`
- THEN the document contains all 8 required top-level fields
- AND `component` equals `"Button"`
- AND `variants` is a non-empty list (e.g., `["primary", "secondary", "ghost", "danger"]`)
- AND `states` contains at least `default`, `hover`, `focus`, `disabled`
- AND `a11yRequirements.role` is `"button"`
- AND `a11yRequirements.focusable` is `true`

#### Scenario: Card spec has allowedChildren

- GIVEN `Card.spec.yaml` is loaded
- WHEN `compositionRules.allowedChildren` is inspected
- THEN it is not empty — Card is a container that accepts child content

### Requirement: Token binding key convention (component tier)

All `tokenBindings` entries in component spec YAMLs SHALL use the W3C DTCG alias format (`{"$value": "{<semantic_token_key>}"}`). Token keys SHALL reference only semantic-tier tokens. No hardcoded values (hex colors, px values, numeric literals) SHALL appear. This is consistent with the primitive spec token binding convention established in p06 and enforced by Token Validation Agent (Agent 8, Token Engine Crew, Phase 2) and Token Compliance Agent (Agent 32, Analytics Crew, Phase 5).

#### Acceptance Criteria

- [ ] Every `tokenBindings` array entry uses DTCG alias format: value is a string matching `\{[a-z.]+\}`
- [ ] No token binding value contains a raw hex color (`#[0-9a-fA-F]{3,8}`)
- [ ] No token binding value contains a CSS length literal (e.g., `"16px"`, `"1rem"`)
- [ ] All token keys use dot-notation referencing semantic-tier namespaces (e.g., `color.interactive.primary.default`, `spacing.component.padding.md`)
- [ ] No global-tier keys appear in any component spec (no `color.blue.500` or similar global primitive references)

#### Scenario: Input spec references semantic color token

- GIVEN `Input.spec.yaml` is generated and loaded
- WHEN the `tokenBindings` array is inspected
- THEN at least one entry references a semantic color token for border styling (e.g., `color.border.input.default`)
- AND the `$value` string starts with `{` and ends with `}`

#### Scenario: No hardcoded values across all component specs

- GIVEN all component specs for `comprehensive` scope are generated
- WHEN every `tokenBindings` list across all 26 specs is inspected
- THEN no `$value` field contains a bare hex color
- AND no `$value` field contains a px or rem literal

### Requirement: Composition from primitives

All component spec `compositionRules.composesFrom` fields SHALL list only the 9 composition primitives (Box, Stack, Grid, Text, Icon, Pressable, Divider, Spacer, ThemeProvider). No component spec SHALL reference another component in `composesFrom`. This enforces the single-level composition contract consumed by Agent 18 (Composition Agent, Component Factory Crew, Phase 3, §4.4).

#### Acceptance Criteria

- [ ] Every component spec has a `compositionRules.composesFrom` array that is non-empty
- [ ] Every value in `composesFrom` is one of: `Box`, `Stack`, `Grid`, `Text`, `Icon`, `Pressable`, `Divider`, `Spacer`, `ThemeProvider`
- [ ] No component name appears in another component's `composesFrom` (no component-to-component composition in specs)
- [ ] `compositionRules.forbiddenNesting` is present (array; may be empty)
- [ ] `compositionRules.slots` is present (array; may be empty — only components with named slot APIs will have entries)

#### Scenario: Button composes from Pressable

- GIVEN `Button.spec.yaml` is loaded
- WHEN `compositionRules.composesFrom` is inspected
- THEN `"Pressable"` is in the list (Button's interactive behaviour is built on Pressable)

#### Scenario: Card composes from Box

- GIVEN `Card.spec.yaml` is loaded
- WHEN `compositionRules.composesFrom` is inspected
- THEN `"Box"` is in the list (Card's layout container is Box)

#### Scenario: Modal has a slot definition

- GIVEN `Modal.spec.yaml` is loaded
- WHEN `compositionRules.slots` is inspected
- THEN at least one slot entry exists (e.g., `header`, `body`, `footer`)

### Requirement: Interactive state machine

Each component spec SHALL declare a `states` object encoding the component's interactive and visual states. All interactive components MUST include at minimum: `default`, `hover`, `focus`, `disabled`. Form components (Input, Checkbox, Radio, Select) MUST additionally include `error` and `success` states. Components with loading or async behavior MUST include `loading`.

#### Acceptance Criteria

- [ ] All Starter-tier interactive components (Button, Input, Checkbox, Radio, Select) have `states` with `default`, `hover`, `focus`, `disabled`
- [ ] Form components (Input, Checkbox, Radio, Select) have `states.error` and `states.success`
- [ ] Non-interactive display components (Avatar, Badge, Card) have at minimum `default` in `states`
- [ ] Alert has a `states` object with at minimum `default`

#### Scenario: Input has error state

- GIVEN `Input.spec.yaml` is loaded
- WHEN `states` is inspected
- THEN keys `default`, `hover`, `focus`, `disabled`, `error`, `success` all exist

#### Scenario: Badge has only default state

- GIVEN `Badge.spec.yaml` is loaded
- WHEN `states` is inspected
- THEN at minimum `default` exists
- AND no `hover` or `focus` state is required (Badge is non-interactive)

### Requirement: Accessibility requirements per component

Every component spec SHALL encode `a11yRequirements` appropriate to its component type. ARIA roles and keyboard interaction patterns SHALL follow WAI-ARIA Authoring Practices. Interactive components SHALL declare keyboard interactions. This data is consumed by Agent 19 (Accessibility Agent, Component Factory Crew, Phase 3, §4.4) to scaffold and validate a11y implementation. Requirements apply at the AA accessibility level by default; the Brand Profile `accessibility` field overrides to AAA.

#### Acceptance Criteria

- [ ] Every component spec has `a11yRequirements.role` (string or `null` for generic containers)
- [ ] Every component spec has `a11yRequirements.focusable` (boolean)
- [ ] Every component spec has `a11yRequirements.keyboardInteractions` (array; non-empty for interactive components)
- [ ] Every component spec has `a11yRequirements.ariaAttributes` (array; may be empty)
- [ ] Button: `role = "button"`, `focusable = true`, `keyboardInteractions` includes Enter and Space
- [ ] Modal: `role = "dialog"`, `focusable = true`, `keyboardInteractions` includes Escape, focus trap documented
- [ ] Select/Dropdown: `role = "combobox"` or `"listbox"`, keyboard interactions include Arrow keys

#### Scenario: Button keyboard interactions

- GIVEN `Button.spec.yaml` is loaded
- WHEN `a11yRequirements.keyboardInteractions` is inspected
- THEN the array contains entries for `Enter` and `Space` key activation

#### Scenario: Modal documents focus trap

- GIVEN `Modal.spec.yaml` is loaded
- WHEN `a11yRequirements` is inspected
- THEN `focusable` is `true`
- AND `keyboardInteractions` contains an entry referencing `Escape` for closing
- AND `ariaAttributes` contains `aria-modal` or `aria-labelledby`

### Requirement: componentOverrides patching

The `generate_component_specs()` function SHALL accept an optional `component_overrides` dict that patches top-level fields in a specific component's spec before serialization. The override MUST be a shallow merge — deep-key replacement of nested sub-fields (e.g., individual props within `props`) is NOT supported. Overrides are consumed from the Brand Profile `componentOverrides` field by Agent 4.

#### Acceptance Criteria

- [ ] Passing `component_overrides={"Button": {"defaultVariant": "secondary"}}` results in `Button.spec.yaml` containing `defaultVariant: secondary`
- [ ] Override values do not affect other component specs
- [ ] Passing `component_overrides={}` or `component_overrides=None` produces the same output as calling without overrides
- [ ] Unknown component keys in `component_overrides` (not present in the scope tier) are silently ignored

#### Scenario: Override default variant for Button

- GIVEN `component_overrides = {"Button": {"defaultVariant": "secondary"}}`
- WHEN `generate_component_specs(scope="starter", component_overrides=component_overrides, output_dir=...)` is called
- THEN `Button.spec.yaml` contains `defaultVariant: secondary`
- AND `Input.spec.yaml` is unaffected

#### Scenario: Override for out-of-scope component is ignored

- GIVEN `component_overrides = {"DataGrid": {"defaultVariant": "compact"}}`
- WHEN `generate_component_specs(scope="starter", ...)` is called (DataGrid is Comprehensive-only)
- THEN no error is raised
- AND 10 Starter spec files are generated without `DataGrid.spec.yaml`

### Requirement: CoreComponentSpecGenerator CrewAI tool

The `CoreComponentSpecGenerator` class SHALL be a `crewai.tools.BaseTool` subclass that wraps `generate_component_specs()`, making it callable by Agent 4 within the DS Bootstrap Crew task flow.

#### Acceptance Criteria

- [ ] `CoreComponentSpecGenerator` is a subclass of `crewai.tools.BaseTool`
- [ ] `CoreComponentSpecGenerator.name == "CoreComponentSpecGenerator"`
- [ ] `CoreComponentSpecGenerator._run(scope, output_dir, component_overrides_json)` returns a non-empty summary string on success (e.g., `"Generated 10 component specs in /path/to/specs"`)
- [ ] `CoreComponentSpecGenerator._run(scope="invalid", ...)` returns an error string (does not raise unhandled exception — BaseTool wraps errors)
- [ ] The tool is importable from `src.daf.tools`

#### Scenario: Tool generates Starter specs

- GIVEN a `CoreComponentSpecGenerator` instance
- WHEN `tool._run(scope="starter", output_dir=str(tmp_path), component_overrides_json="{}")` is called
- THEN 10 YAML files exist under `tmp_path/specs/`
- AND the return value is a non-empty string containing `"10"` and the output path

### Requirement: Agent 4 module structure

The `create_core_component_agent()` factory SHALL return a `crewai.Agent` configured for the Core Component Agent role. The `create_core_component_task()` factory SHALL return a `crewai.Task` wiring Task T4 to AGENT 4. Both SHALL follow the module pattern established by `agents/token_foundation.py` (Agent 2) and `agents/primitive_scaffolding.py` (Agent 3).

#### Acceptance Criteria

- [ ] `create_core_component_agent()` returns a `crewai.Agent` instance
- [ ] The agent's `role` is `"Core Component Agent"`
- [ ] `CoreComponentSpecGenerator` is in the agent's `tools` list
- [ ] The agent uses a Tier 2 model: `os.environ.get("DAF_TIER2_MODEL", "claude-sonnet-4-20250514")`
- [ ] `create_core_component_task()` returns a `crewai.Task` instance
- [ ] `create_core_component_agent` and `create_core_component_task` are exported from `src.daf.agents`

#### Scenario: Agent factory produces valid Agent

- GIVEN `from daf.agents import create_core_component_agent`
- WHEN `create_core_component_agent()` is called
- THEN the result is a `crewai.Agent` instance
- AND `CoreComponentSpecGenerator` is accessible in `agent.tools`
