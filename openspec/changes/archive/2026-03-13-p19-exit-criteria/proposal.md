# Proposal: p19-exit-criteria

## Intent

PRD §8 defines 15 explicit exit criteria (8 Fatal + 7 Warning) that determine whether a generated design system is valid and complete. Currently, these checks are scattered across individual tools (`dtcg_schema_validator.py`, `contrast_safe_pairer.py`, `token_ref_checker.py`, `eslint_runner.py`, `gate_evaluator.py`, etc.) with no unified evaluator that runs all 15 checks in a single pass, classifies results by severity, and writes a canonical `reports/exit-criteria.json` report.

The pipeline sets `final_status` in the Release Crew based only on npm/tsc/test outcomes from `TestResultParser` — a subset of the 8 Fatal criteria. The remaining Fatal checks (DTCG schema validity, token reference resolution, WCAG contrast, CSS custom property resolution) are evaluated by individual crew agents but their results are not aggregated into a single pass/fail signal that gates pipeline completion. As a result, `isComplete: true` can be written even when Fatal checks 1–6 have failed silently.

This change introduces a dedicated `ExitCriteriaEvaluator` tool and a corresponding `ExitCriteriaAgent` (Agent 30-bis, Governance Crew) that runs all 15 §8 checks against the output directory, produces a structured report, and enforces the correct `isComplete` value in `reports/generation-summary.json`.

## Scope

### In scope

- Implement **`ExitCriteriaEvaluator`** (`src/daf/tools/exit_criteria_evaluator.py`): a `BaseTool` that takes the output directory and invokes each of the 15 §8 checks by delegating to the relevant existing tools. Returns a structured `ExitCriteriaReport` with per-criterion `id`, `description`, `severity` (`fatal` | `warning`), `passed` (bool), and `detail` (error message or empty string).
- Implement the 8 Fatal criterion runners as private helper functions inside `exit_criteria_evaluator.py`:
  - C1: `_check_token_json_parses` — `json.loads` on `tokens/` JSON files
  - C2: `_check_dtcg_schema` — delegates to `DtcgSchemaValidator`
  - C3: `_check_semantic_refs_resolve` — delegates to `TokenGraphTraverser`
  - C4: `_check_component_refs_resolve` — delegates to `TokenGraphTraverser`
  - C5: `_check_wcag_contrast` — delegates to `ContrastSafePairer`
  - C6: `_check_css_custom_props` — delegates to `TokenRefChecker`
  - C7: `_check_tsc_no_errors` — delegates to `EslintRunner` (tsc subprocess)
  - C8: `_check_npm_build` — delegates to `DependencyResolver`
- Implement the 7 Warning criterion runners:
  - C9: `_check_unit_tests_pass` — reads `reports/generation-summary.json` for test results
  - C10: `_check_no_hardcoded_values` — delegates to `AstPatternMatcher`
  - C11: `_check_aria_roles` — delegates to `A11yAttributeExtractor`
  - C12: `_check_quality_scores` — reads `reports/governance/quality-gates.json` via `GateStatusReader`
  - C13: `_check_drift_consistency` — reads drift report from `reports/governance/drift-report.json`
  - C14: `_check_registry_valid` — reads and validates `reports/component-registry.json` via `JsonSchemaValidator`
  - C15: `_check_no_failed_components` — reads `reports/generation-summary.json`
- Write `reports/exit-criteria.json` with the full 15-criterion report and a top-level `isComplete` bool (true iff all Fatal criteria pass).
- Implement **`ExitCriteriaAgent`** (`src/daf/agents/exit_criteria.py`): Agent 30-bis factory that uses `ExitCriteriaEvaluator` and `ReportWriter`. Wired into the Governance Crew after Agent 30 (Quality Gate Agent) as task T5-bis.
- Update **`src/daf/crews/governance.py`** to add the `ExitCriteriaAgent` task after `t4_quality_gate`, passing its `reports/exit-criteria.json` output to the subsequent pipeline summary.
- Update **`src/daf/crews/release.py`** task `t6_final_status` to read `reports/exit-criteria.json` and derive `final_status` and `isComplete` from its `isComplete` field rather than re-evaluating.
- Full unit-test coverage for `ExitCriteriaEvaluator` and `ExitCriteriaAgent`, following TDD (tests written first, red→green cycle).

### Out of scope

- Changes to individual tool implementations (`DtcgSchemaValidator`, `ContrastSafePairer`, `TokenRefChecker`, etc.) — they are called as-is.
- Changes to `GateEvaluator` or the 5-gate per-component evaluation.
- Changes to the 4 TypeScript test suites (`tokens.test.ts`, `a11y.test.ts`, `composition.test.ts`, `compliance.test.ts`) generated by `TestSuiteGenerator`.
- Changes to the WCAG contrast threshold value or quality score threshold — these remain at their PRD-specified values.
- Changes to Human Gate 1 or Human Gate 2 review flows.
- Changes to any other crew (Token Engine, Design-to-Code, Component Factory, Documentation, Analytics, AI Semantic Layer).

## Approach

1. Implement `ExitCriteriaEvaluator` as a `BaseTool` that delegates to existing tools for each of the 15 criteria. Each helper function returns a `CriterionResult(id, description, severity, passed, detail)` dataclass.
2. The evaluator aggregates results, determines `isComplete` (True iff all Fatal criteria pass), and writes `reports/exit-criteria.json`.
3. `ExitCriteriaAgent` wraps the evaluator with a Haiku-tier LLM. Its single task receives the output directory and writes the report.
4. The Governance Crew adds this task after Agent 30's `t4_quality_gate` task.
5. The Release Crew's `t6_final_status` task reads `reports/exit-criteria.json` and uses its `isComplete` to set `final_status: success | partial | failed`.

## Risks

- **Tool interface mismatch:** Several delegate tools (`DtcgSchemaValidator`, `ContrastSafePairer`, `TokenRefChecker`) were designed for agent-driven invocation. Their return shapes must be normalised into pass/fail booleans inside the evaluator — this requires careful inspection of each tool's `_run` return value.
- **Missing output files:** If an upstream phase failed and didn't produce expected files (e.g., no `tokens/` directory), individual criterion runners must handle `FileNotFoundError` gracefully and return `passed=False` with an informative detail rather than raising.
- **tsc/npm as subprocesses:** C7 and C8 invoke subprocesses. In CI and unit-test environments these must be mocked or the tests will be slow/flaky.
