# Specification: ThemeProvider Primitive

## Purpose

Defines the API contract, behavior, and generation requirements for the `ThemeProvider` React primitive and its companion `useTheme()` hook. `ThemeProvider` is one of the 9 base primitives generated by the Design-to-Code Crew (Phase 3) for every design system produced by DAF.

Affected crew: Design-to-Code Crew (Agents 11–15), DS Bootstrap Crew (Agent 4 — Primitive Spec Generator).

---

## Requirements

### Requirement: ThemeProvider Component Contract

The generated `ThemeProvider.tsx` SHALL be a React context provider that manages the active theme and brand for the design system. It MUST apply the active theme and brand as CSS classes on the root element of the DOM.

The component is theme-agnostic with respect to its own visual styling — it renders no visible UI and consumes no design tokens directly.

#### Acceptance Criteria

- [ ] `ThemeProvider` renders a `React.Fragment` (no DOM wrapper); theme classes are applied to `document.documentElement`
- [ ] Accepts a `defaultTheme` prop of type `string`; defaults to OS `prefers-color-scheme` preference if a matching theme name exists in `availableThemes`, otherwise defaults to `availableThemes[0]`
- [ ] Accepts a `defaultBrand` prop of type `string | null`; defaults to `null` (no brand class applied)
- [ ] Accepts `availableThemes: string[]` and `availableBrands: string[]` props that define the valid values for theme and brand switching
- [ ] Applies exactly one `theme-<name>` class to `document.documentElement` at any time; removes the previous theme class before applying the new one
- [ ] When `brand` is non-null, applies a `brand-<name>` class alongside the theme class; removes previous brand class on change
- [ ] When `brand` is null, no `brand-*` class is present on `document.documentElement`
- [ ] Exposes theme state via a React context consumed by `useTheme()`
- [ ] Cleans up the `prefers-color-scheme` `MediaQueryList` event listener on component unmount
- [ ] Throws a runtime error (development mode only) when `defaultTheme` is not in `availableThemes`
- [ ] Is generated as a client component (no `"use server"` directive)

#### Scenario: Default theme applied on mount

- **WHEN** `<ThemeProvider defaultTheme="light" availableThemes={["light", "dark"]} />` mounts
- **THEN** `document.documentElement.classList` contains `"theme-light"`
- **AND** no `brand-*` class is present
- **AND** the class is applied synchronously before the first render of children

#### Scenario: Theme switch removes previous class

- **WHEN** the active theme is `"light"` and `setTheme("dark")` is called
- **THEN** `"theme-light"` is removed from `document.documentElement.classList`
- **AND** `"theme-dark"` is added
- **AND** no other `theme-*` classes remain

#### Scenario: Brand class applied alongside theme class

- **WHEN** the active theme is `"dark"` and `setBrand("brand-a")` is called
- **THEN** `document.documentElement.classList` contains both `"theme-dark"` and `"brand-a"`

#### Scenario: Brand set to null removes brand class

- **WHEN** the active brand is `"brand-a"` and `setBrand(null)` is called
- **THEN** `"brand-a"` is removed from `document.documentElement.classList`
- **AND** the active theme class remains unchanged

#### Scenario: OS dark mode preference detected

- **WHEN** `defaultTheme` prop is not provided
- **AND** `window.matchMedia("(prefers-color-scheme: dark)").matches` is `true`
- **AND** `"dark"` is in `availableThemes`
- **THEN** the initial theme is `"dark"` and `"theme-dark"` is applied on mount

#### Scenario: OS preference mismatch falls back to first available theme

- **WHEN** `defaultTheme` prop is not provided
- **AND** OS reports dark preference
- **AND** `availableThemes` is `["light"]` (dark not available)
- **THEN** the initial theme is `"light"` (the first available)

#### Scenario: MediaQueryList listener is cleaned up

- **WHEN** the `ThemeProvider` component unmounts
- **THEN** the `change` event listener on the `prefers-color-scheme` `MediaQueryList` is removed
- **AND** no further OS theme-change events affect application state

### Requirement: useTheme Hook Contract

The `useTheme()` hook SHALL return the current theme state and setter functions from the nearest `ThemeProvider` in the React tree.

#### Acceptance Criteria

- [ ] Returns an object matching the `ThemeContextValue` interface: `{ theme, brand, setTheme, setBrand, availableThemes, availableBrands }`
- [ ] `theme` is a non-null `string` at all times (always has an active theme)
- [ ] `brand` is `string | null` (null means no brand override active)
- [ ] `setTheme(name: string)` is a stable function reference (does not change between renders unless `availableThemes` changes)
- [ ] `setBrand(name: string | null)` is a stable function reference
- [ ] Calling `setTheme` with a value not in `availableThemes` throws in development mode and is a no-op in production
- [ ] Calling `useTheme()` outside a `ThemeProvider` throws a descriptive error: `"useTheme must be used within a ThemeProvider"`

#### Scenario: Reading theme inside a provider

- **WHEN** a component calls `useTheme()` inside a `ThemeProvider` with `defaultTheme="light"`
- **THEN** `theme` equals `"light"` and `brand` equals `null` on initial render

#### Scenario: Programmatic theme change updates hook consumers

- **WHEN** `setTheme("dark")` is called from any component within the provider tree
- **THEN** all components consuming `useTheme()` re-render with `theme === "dark"`

#### Scenario: useTheme outside provider throws

- **WHEN** a component calls `useTheme()` with no `ThemeProvider` ancestor
- **THEN** a descriptive error is thrown identifying the missing provider

### Requirement: ThemeProvider Spec YAML Generation

The Primitive Spec Generator Agent (4) MUST produce a `ThemeProvider.spec.yaml` file that the Design-to-Code Crew (Phase 3) uses as the canonical source for generating `ThemeProvider.tsx` and `useTheme.ts`.

#### Acceptance Criteria

- [ ] `ThemeProvider.spec.yaml` defines the component props: `defaultTheme`, `defaultBrand`, `availableThemes`, `availableBrands`, `children`
- [ ] The spec declares that `ThemeProvider` consumes no design tokens (it applies theme classes, not token values)
- [ ] The spec references `useTheme` as an exported companion hook with its full `ThemeContextValue` type interface
- [ ] The spec is written to `specs/ThemeProvider.spec.yaml` in the output folder

#### Scenario: Spec YAML contains required fields

- **WHEN** the Primitive Spec Generator (Agent 4) runs for the ThemeProvider
- **THEN** `specs/ThemeProvider.spec.yaml` exists in the output folder
- **AND** it contains a `props` section with all five required props
- **AND** it contains a `tokenBindings: []` entry (empty — no token consumption)
- **AND** it contains an `exports: ["ThemeProvider", "useTheme"]` declaration
